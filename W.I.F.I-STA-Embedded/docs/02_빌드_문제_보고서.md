# 빌드 문제 해결 보고서

- **대상 프로젝트**: `W.I.F.I-STA-Embedded` (ESP-IDF v5.5.4 / 타겟 esp32)
- **환경**: Windows 11, VSCode + ESP-IDF 확장 2.1.0, IDF 설치 위치 `C:\Espressif`
- **최종 결과**: 클린 빌드 성공 (종료코드 0, 경고·에러 0건). Windows PowerShell 5.1 환경에서도 빌드 재현 확인.

> 빌드 문제는 성격이 다른 **두 층**으로 나뉜다.
> **A**는 "프로젝트가 컴파일되지 않는" 코드/구성 문제, **B**는 "프로젝트는 멀쩡한데 빌드를
> 실행(launch)하지 못하는" 환경 문제. 처음엔 둘이 섞여 보였으나 A를 먼저 해결한 뒤
> 실제 컴파일은 성공했고, 이후 남은 증상은 전부 B(실행 환경)였다.

---

## A. 코드/구성 단계 빌드 차단 요소

STA→AP 전환 후, 프로젝트에 남아 있던 AT(송신기) 잔재와 미정의 함수 때문에
실제 빌드가 불가능한 상태였다. 아래 5건을 수정했다.

| # | 문제 | 원인 | 조치 |
|---|------|------|------|
| 1 | `main/CMakeLists.txt`가 컴포넌트 `espnowsta` 요구 | 실제 폴더명은 `espnowap` — 이름 불일치로 컴포넌트 미해결 | `PRIV_REQUIRES`를 `espnowap`으로 수정 |
| 2 | `espnowap`에 AP용 `wifiInit()` 중복 정의 | `wifimodule/wifi.c`의 `wifiInit()`과 심볼 충돌(링크 에러) | `espnowap`에서 `wifiInit`/`wifiHandler`/`espnow_csi_send`(AT 잔재) 제거 |
| 3 | `espnowap`이 `PIR`, `APcommon` 컴포넌트 의존 | STA 프로젝트에는 두 컴포넌트가 없음 → CMake 구성 실패 | `CMakeLists.txt`의 `REQUIRES`를 `esp_wifi log`로 정리, 관련 헤더 의존 제거 |
| 4 | `espnow_init_setup()` / `espnow_send_pairing_request()` 미정의 | `main.c`에서 호출하나 정의가 어디에도 없음 | STA(AP) 수신기용으로 신규 구현 (ESP-NOW init + 브로드캐스트 피어 등록 + 페어링 요청 송신) |
| 5 | 부팅 시 `esp_wifi_set_csi()`에서 패닉 위험 | `sdkconfig`에 CSI 비활성(`CONFIG_..._CSI_ENABLED is not set`) | `CONFIG_ESP_WIFI_CSI_ENABLED=y`, `CONFIG_ESP32_WIFI_CSI_ENABLED=y` 활성화 |

> #5는 컴파일은 통과하지만 런타임에서 `ESP_ERROR_CHECK`가 abort를 유발하는 사항으로,
> "빌드는 되지만 부팅 즉시 죽는" 문제를 예방하기 위해 함께 처리했다.

### 수정 파일 목록 (A)
- `main/CMakeLists.txt`
- `components/espnowap/espnowAP.c` (재작성)
- `components/espnowap/espnowAP.h` (재작성)
- `components/espnowap/CMakeLists.txt`
- `sdkconfig`

**→ A 처리 후 CLI 빌드가 종료코드 0으로 성공. 프로젝트 코드는 정상임이 확정됨.**

## B. 빌드 실행 환경 문제 (사용자 측)

코드는 정상인데 "빌드가 안 된다"고 보인 별개의 실행/환경 이슈들. 빌드 실패가 아니라
**빌드 명령이 시작조차 안 된** 경우가 대부분이었다.

### B-1. `idf.py: not recognized` (일반 PowerShell / VSCode 기본 터미널)
- **증상**: 터미널에서 `idf.py build` → "term 'idf.py' is not recognized"
- **원인**: `idf.py`는 전역 실행파일이 아니라 **ESP-IDF 환경을 활성화해야 등록되는
  명령(함수)**. 활성화 = PATH에 컴파일러/ninja/python venv/`IDF_PATH` 등 일괄 등록.
  VSCode 기본 통합 터미널은 일반 PowerShell이라 이게 안 잡혀 있음.
- **해결**: 다음 중 하나
  - VSCode **ESP-IDF Terminal** 사용: `Ctrl+Shift+P` → `ESP-IDF: Open ESP-IDF Terminal`
    (이 터미널은 이미 활성화돼 있어 `idf.py build`가 바로 동작)
  - 또는 일반 터미널에서 먼저 활성화 후 실행 (그 창에서만 유효):
    ```powershell
    $env:IDF_PATH="C:\Espressif\frameworks\esp-idf-v5.5.4"; . C:\Espressif\Initialize-Idf.ps1
    idf.py build
    ```
  - 또는 헬퍼 스크립트 `build.ps1` 사용 (B-3 참조)

### B-2. 빌드 캐시 ↔ 파이썬 환경 불일치 (하단 🔨 버튼 빌드 에러, 세부 미표시)
- **증상**: VSCode 하단 빌드 버튼 클릭 시 에러가 나는데 상세 내용이 안 보임
- **원인**: IDF가 두 파이썬 venv 보유 (`idf5.5_py3.11_env`, `idf5.5_py3.13_env`).
  - 초기 CLI 빌드가 **py3.13**으로 `build/` 캐시 생성
  - VSCode 설정(`idf.pythonBinPathWin`)은 **py3.11** 지정
  - → `build/`의 CMake 캐시와 버튼이 쓰는 파이썬 경로 불일치로 재구성 실패.
    버튼은 에러 상세를 알림으로만 띄워 원인 파악이 어려웠음.
- **해결**:
  1. 기존 `build/` 폴더 삭제(생성물, 안전)
  2. `.vscode/settings.json`의 `idf.pythonBinPathWin`을 **py3.13**으로 통일
     (`idf5.5_py3.11_env` → `idf5.5_py3.13_env`) → 버튼/터미널/CLI 캐시 일치
  3. **VSCode 창 새로고침**(`Developer: Reload Window`) 후 버튼 재시도
- **버튼 에러 상세 보는 법**: `보기 → 출력`(`Ctrl+Shift+U`) → 우측 드롭다운에서 **`ESP-IDF`**
  선택 시 실제 빌드 로그가 출력됨.

### B-3. 헬퍼 스크립트 `build.ps1` 작성 중 발생한 문제들
사용자가 활성화를 매번 신경 쓰지 않도록 원클릭 `build.ps1`을 제공. 작성 과정에서 두 번 실패.

| 시도 | 실패 원인 | 조치 |
|------|-----------|------|
| 1차 | 스크립트 내 **한글 메시지**를 Windows PowerShell 5.1이 깨뜨림(인코딩) → `The string is missing the terminator` 파싱 에러 | 모든 메시지를 **영문(ASCII)**으로 교체 |
| 2차 | `Initialize-Idf.ps1`이 PS 5.1에서 `idf-env`로 파이썬을 못 찾음(`$PythonCommand` = null → `Get-Item ...\null` 에러) | 활성화를 **ESP-IDF 공식 `export.ps1` 직접 호출**로 변경하고, **venv 파이썬(py3.13)을 PATH 앞에 명시적으로 추가** |

- **최종 `build.ps1` 동작**: `IDF_PATH`/`IDF_TOOLS_PATH` 설정 → venv `py3.13\Scripts`를 PATH
  선두에 추가 → `export.ps1` 닷-소싱(활성화) → `idf.py build`.
- **검증**: 사용자와 동일한 `powershell -ExecutionPolicy Bypass -File .\build.ps1` 방식으로
  **Windows PowerShell 5.1에서 `Project build complete.` 확인**.
- **사용법**:
  ```powershell
  powershell -ExecutionPolicy Bypass -File "C:\project\wify\W.I.F.I-STA-Embedded\W.I.F.I-STA-Embedded\build.ps1"
  ```
  플래시까지: 끝에 ` -Flash -Port COM5` (포트는 보드에 맞게), 클린 빌드: ` -Clean`

### 수정/추가 파일 목록 (B)
- `.vscode/settings.json` (파이썬 env py3.11 → py3.13)
- `build.ps1` (신규, 활성화+빌드 원클릭 헬퍼)

## C. 빌드 방법별 비교

| 방법 | 환경 활성화 | 비고 |
|------|------------|------|
| 일반 터미널 `idf.py build` | ❌ 수동 활성화 필요 | 활성화 없이는 "not recognized" |
| VSCode **ESP-IDF Terminal** + `idf.py build` | ✅ 자동 | 권장. `Ctrl+Shift+P` → ESP-IDF: Open ESP-IDF Terminal |
| 하단 🔨 **버튼** | ✅ 확장이 자동 | 캐시/파이썬 통일 + 창 새로고침 후 동작. 에러는 Output→ESP-IDF에서 확인 |
| **`build.ps1`** | ✅ 스크립트가 대행 | 검증 완료. 일반 터미널에서 한 줄로 실행 가능 |

## E. 향후 주의사항

- **파이썬 env 통일 유지**: IDF 설치/업데이트 시 venv가 늘 수 있음. CLI·VSCode·스크립트가
  **같은 파이썬 env(py3.13)**를 쓰도록 맞춰야 캐시 충돌이 없다.
- **캐시 충돌 시 1차 조치**: VSCode `ESP-IDF: Full Clean` 또는 `build/` 폴더 삭제 후 재빌드.
- **일반 터미널 빌드 시**: 반드시 환경 활성화(또는 ESP-IDF Terminal/`build.ps1`) 후 `idf.py` 사용.
- **PowerShell 스크립트 인코딩**: Windows PowerShell 5.1과 함께 쓸 `.ps1`은 한글(비ASCII)을
  피하거나 UTF-8 BOM으로 저장. 본 프로젝트의 `build.ps1`은 ASCII 전용으로 유지.
- #4의 `espnow_init_setup`/`espnow_send_pairing_request`는 빌드 통과용으로 신규 작성된
  것이므로, 상대 기기와의 **실제 페어링 프로토콜/페이로드 형식**은 실기에서 검증·정합 필요.
