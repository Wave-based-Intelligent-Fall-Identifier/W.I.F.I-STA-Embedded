# 기술 변경 상세 보고서 (What / How / Why)

- **대상**: `W.I.F.I-STA-Embedded` (ESP-IDF v5.5.4, esp32)
- **목표**: 수신기(STA) 기기의 WiFi 네트워킹 역할을 **STA → AP**로 전환하고,
  이로 인해 드러난 빌드 차단 요소를 정리하여 빌드 가능 상태로 만든다.
- **불변 조건**: CSI 수신·계산·baseline 등 기능 로직은 변경하지 않는다.

---

## 0. 시스템 맥락 (변경의 전제)

WiFi CSI 센싱은 **송신기**가 패킷을 계속 흘리고, **수신기**가 그 패킷의
채널 상태 정보(CSI)를 측정하는 구조다. 본 작업은 두 기기의 네트워킹 역할을 맞바꾸는
과정의 "수신기(STA 프로젝트)" 파트로, **이 기기를 STA(접속) 대신 AP(접속점)로 동작**시킨다.

핵심 통찰: **CSI 캡처는 `esp_wifi_set_promiscuous(true)` 기반이라 WiFi 모드(STA/AP)와
무관하게 동작한다.** 따라서 CSI 파이프라인은 손대지 않고 WiFi 초기화 계층만 교체하면 된다.
(esp_wifi_set_promiscuous(true) : esp32에서 wifi 자신을 제외한 주변 기기에만 패킷을 흘리도록 프록미스큐어스 모드를 활성화 시키는 것)

## 1. `components/wifimodule/wifi.c` — STA → AP 전환

### 1-1. 네트워크 인터페이스 생성
**What**
```c
- esp_netif_create_default_wifi_sta();
+ esp_netif_create_default_wifi_ap();
```
**Why**: `esp_netif`는 모드별로 다른 기본 인터페이스 객체를 만든다. STA용 함수는
DHCP 클라이언트가 붙은 station netif를, AP용 함수는 DHCP 서버가 붙은 softAP netif를
생성한다. AP로 동작하려면 반드시 AP netif여야 이후 `WIFI_IF_AP` 설정이 유효하다.

### 1-2. 이벤트 핸들러 `wifiHandler()`
**What** — STA 생명주기 이벤트를 AP 이벤트로 교체
```c
// Before (STA): 연결 시도 → 연결됨 → 끊김 시 재시도 → IP 획득
case WIFI_EVENT_STA_START:        esp_wifi_connect();              ...
case WIFI_EVENT_STA_CONNECTED:    xEventGroupSetBits(...CONNECTED) ...
case WIFI_EVENT_STA_DISCONNECTED: 재시도 / FAIL_BIT                ...
case IP_EVENT_STA_GOT_IP:         xEventGroupSetBits(...GOT_IP)    ...

// After (AP): AP 시작 / 클라이언트 접속·해제 로깅
if      (eventId == WIFI_EVENT_AP_START)          ESP_LOGI("AP 시작");
else if (eventId == WIFI_EVENT_AP_STACONNECTED)   ESP_LOGI("장치 접속 " MACSTR ...);
else if (eventId == WIFI_EVENT_AP_STADISCONNECTED)ESP_LOGI("장치 해제 " MACSTR ...);
```
**Why**:
- STA의 동작 모델(능동적으로 AP에 *접속*하고 IP를 받는다)과 AP의 동작 모델(수동적으로
  클라이언트의 *접속을 받는다*)은 이벤트 자체가 다르다. AP는 `esp_wifi_connect()`,
  연결 재시도, DHCP IP 획득 개념이 없다 → 관련 분기를 전부 제거.
- AP는 "누가 접속/해제했는가"가 주요 이벤트라 `AP_STACONNECTED/STADISCONNECTED`로 대체.

### 1-3. WiFi 설정 구조체 (`.sta` → `.ap`)
**What**
```c
// Before
wifi_config_t wifi_config = {0};
strcpy((char *)wifi_config.sta.ssid, WIFI_SSID);
strcpy((char *)wifi_config.sta.password, WIFI_PASS);

// After
wifi_config_t wifi_config = {
    .ap = {
        .ssid           = WIFI_SSID,
        .ssid_len       = strlen(WIFI_SSID),
        .channel        = 6,
        .password       = WIFI_PASS,
        .max_connection = 2,
        .authmode       = WIFI_AUTH_WPA_WPA2_PSK,
    },
};
if (strlen(WIFI_PASS) == 0) {
    wifi_config.ap.authmode = WIFI_AUTH_OPEN;
}
```
**Why**:
- `wifi_config_t`는 union이다. STA일 땐 `.sta`(접속 대상 SSID/PW)를, AP일 땐
  `.ap`(내가 송출할 SSID/PW + 운영 파라미터)를 채워야 한다.
- AP 전용 필드 의미: `channel`(송출 채널, CSI 송수신 채널과 일치 필요), `max_connection`
  (동시 접속 허용 수), `authmode`(보안 방식), `ssid_len`(명시적 길이 지정).
- 비밀번호가 비면 WPA2가 성립할 수 없으므로 `WIFI_AUTH_OPEN`으로 강등 — 빈 PW로 AP를
  올릴 때 흔한 가드.

### 1-4. 모드·인터페이스 지정
**What**
```c
- esp_wifi_set_mode(WIFI_MODE_STA);
- esp_wifi_set_config(WIFI_IF_STA, &wifi_config);
+ esp_wifi_set_mode(WIFI_MODE_AP);
+ esp_wifi_set_config(WIFI_IF_AP, &wifi_config);
```
**Why**: 모드와 설정 인터페이스(`WIFI_IF_*`)가 `.sta`/`.ap` 선택과 일치해야 한다.
불일치 시 설정이 무시되거나 오류가 난다.

### 1-5. IP 획득 대기 로직 제거
**What**
```c
// Before: GOT_IP / FAIL 비트를 무한 대기 후 분기
EventBits_t bits = xEventGroupWaitBits(wifiEventGroup, GOT_IP_BIT | FAIL_BIT, ...);
if (bits & GOT_IP_BIT) return ESP_OK; else ...

// After: AP는 IP를 받지 않으므로 시작 직후 반환
ESP_LOGI(TAG, "WiFi(AP) 초기화 성공");
return ESP_OK;
```
**Why**: STA는 "IP를 받아야 통신 준비 완료"라 GOT_IP까지 블로킹했지만, AP는 DHCP
*서버*이므로 스스로 IP를 받는 단계가 없다. 그대로 두면 영원히 오지 않을 GOT_IP를
기다리며 `wifiInit()`이 반환하지 못한다.

### 1-6. 자기 MAC 조회 인터페이스
**What**
```c
- esp_wifi_get_mac(WIFI_IF_STA, mac);
+ esp_wifi_get_mac(WIFI_IF_AP, mac);
```
**Why**: 이제 활성 인터페이스가 AP다. 송신기 측 CSI 필터가 이 기기의 MAC을 기준으로
삼을 수 있으므로, 실제 동작 인터페이스의 MAC을 출력하는 것이 의미상 정확하다.

### 1-7. 미사용 변수 제거
**What**: `static uint8_t retryCounts = 0;` 삭제.
**Why**: STA 끊김 재시도 카운터였고, 해당 분기를 제거하면서 미사용이 됨 → 불필요한
경고 유발 가능성 제거(코드 위생).

## 2. `components/wifimodule/wifi.h` — 헤더 인클루드 추가
**What**
```c
+ #include "esp_mac.h"
```
**Why**: AP 이벤트 로깅에 쓰는 `MACSTR` / `MAC2STR` 매크로가 `esp_mac.h`에 정의돼 있다.
기존 STA 코드엔 해당 매크로 사용처가 없어 인클루드가 빠져 있었다.

## 3. `components/espnowap` — AT 잔재 제거 및 STA(AP) 수신기용 재작성

이 컴포넌트는 AT(송신기) 코드가 그대로 복사된 잔재였고, 빌드를 막는 핵심 원인이었다.

### 3-1. 중복 `wifiInit()` 제거 (링크 충돌)
**What**: `espnowAP.c`에 있던 `wifiInit()`, `wifiHandler()`, `espnow_csi_send()` 삭제.
**Why**: `wifimodule/wifi.c`에도 외부 링키지 심볼 `wifiInit`이 존재한다. C에서 동일
이름의 비-static 전역 함수가 두 번역 단위에 정의되면 링커가 *multiple definition*으로
실패한다. WiFi 초기화의 단일 소유권을 `wifimodule`에 두고 중복을 제거.

### 3-2. 불필요 의존 제거
**What**: `pir-sensor.h`, `APconfig.h`, `nowMutex`, `espnow_payload_t` 의존 제거.
**Why**: 이들은 AT 프로젝트의 `PIR`/`APcommon` 컴포넌트 소속인데 STA 프로젝트엔 그
컴포넌트가 없다. 남겨두면 헤더 미해결 및 CMake `REQUIRES` 실패로 이어진다.

### 3-3. `main.c`가 호출하지만 정의가 없던 두 함수 구현
**What**
```c
esp_err_t espnow_init_setup(void) {
    esp_now_init();
    esp_now_peer_info_t peer = {0};
    memcpy(peer.peer_addr, BROADCAST_MAC, 6);
    peer.channel = 0;          // 0 = 현재 WiFi 채널 사용
    peer.ifidx   = WIFI_IF_AP; // AP 인터페이스로 송신
    peer.encrypt = false;
    esp_now_add_peer(&peer);
    ...
}
esp_err_t espnow_send_pairing_request(void) {
    uint8_t payload = 0;
    return esp_now_send(BROADCAST_MAC, &payload, sizeof(payload));
}
```
**Why / 설계 근거**:
- `esp_now_init()`은 WiFi가 start된 뒤 호출해야 하며, `main.c`에서 `wifiInit()` 다음에
  호출되므로 순서가 보장된다.
- ESP-NOW 송신은 대상 MAC을 peer로 등록해야 한다. 상대 MAC을 모르는 초기 페어링
  단계라 **브로드캐스트(FF:FF:FF:FF:FF:FF)** peer를 등록한다.
- `peer.channel = 0`: 별도 채널 고정 없이 현재 WiFi 채널(=AP 채널 6)을 사용 → 채널
  불일치로 인한 송신 실패 방지.
- `peer.ifidx = WIFI_IF_AP`: 이 기기가 AP이므로 AP 인터페이스로 송신해야 한다.
- `espnow_send_pairing_request()`는 자신의 존재를 브로드캐스트로 알리는 최소 신호.
  > 주의: 페이로드 형식은 빌드 통과 및 기본 동작용 최소 구현이다. 상대 기기와의 실제
  > 페어링 프로토콜이 정해져 있다면 페이로드 구조를 그쪽에 맞춰야 한다.

## 4. CMake 구성 정합

### 4-1. `main/CMakeLists.txt`
**What**
```cmake
- PRIV_REQUIRES wifimodule espnowsta baseline
+ PRIV_REQUIRES wifimodule espnowap  baseline
```
**Why**: ESP-IDF에서 컴포넌트 이름은 기본적으로 **폴더명**이다. 실제 폴더는 `espnowap`인데
`espnowsta`를 요구해 컴포넌트를 찾지 못했다 → 실재하는 이름으로 정정.

### 4-2. `components/espnowap/CMakeLists.txt`
**What**
```cmake
- REQUIRES freertos esp_wifi log nvs_flash esp_event PIR APcommon
+ REQUIRES esp_wifi log
```
**Why**: 3-2에서 PIR/APcommon 의존을 끊었으므로 해당 `REQUIRES`도 제거. ESP-NOW
API(`esp_now_*`)는 `esp_wifi` 컴포넌트가 제공하고, 로깅은 `log`가 제공한다 → 최소 의존만 유지.

## 5. `sdkconfig` — CSI 기능 활성화
**What**
```diff
- # CONFIG_ESP_WIFI_CSI_ENABLED is not set
+ CONFIG_ESP_WIFI_CSI_ENABLED=y
- # CONFIG_ESP32_WIFI_CSI_ENABLED is not set
+ CONFIG_ESP32_WIFI_CSI_ENABLED=y
```
**Why**:
- CSI 수신은 컴파일 타임 옵션으로 보호된다. 비활성 상태에서는 `esp_wifi_set_csi(true)`가
  `ESP_ERR_NOT_SUPPORTED`를 반환한다. `main.c`가 이를 `ESP_ERROR_CHECK`로 감싸므로
  **부팅 직후 abort(패닉)**가 발생한다. → "빌드는 되지만 즉시 죽는" 문제.
- esp32 타겟은 `CONFIG_ESP32_WIFI_CSI_ENABLED`(칩 계열), 공통 레이어는
  `CONFIG_ESP_WIFI_CSI_ENABLED`로 노출되어 둘 다 켠다.

## 6. 변경하지 않은 것 (의도적 유지)와 근거

| 유지 대상 | 근거 |
|-----------|------|
| `csi_callback()`, `csi_data_calculate()` | CSI 측정·진폭 계산 로직. 모드 무관하게 동작 |
| `esp_wifi_set_promiscuous(true)` 흐름(`main.c`) | CSI 캡처의 본질. AP 모드에서도 그대로 유효 |
| `TX_MAC_ADDRESS` 송신기 필터 | 기능 정책. 단 실기에서 실제 상대 MAC과 일치 확인 필요 |
| baseline 필터 호출부 | 기능 로직, 변경 불필요 |
| `esp_wifi_set_channel(6)`(`main.c`) | AP `.ap.channel = 6`과 일치시켜 충돌 없음 |

## 7. 검증

- **빌드**: ESP-IDF v5.5.4 / esp32, 종료코드 0, 관련 warning·error 0건.
  산출물 `build/W.I.F.I-STA-Embedded.bin`(0xb8c40 bytes, 앱 파티션 28% 여유).
- **환경 재현성**: pwsh 7 및 Windows PowerShell 5.1 양쪽에서 빌드 성공 확인.
- **미검증(실기 필요)**: 상대 기기와의 ESP-NOW 페어링·CSI 실수신 동작, `TX_MAC_ADDRESS`
  실값 일치 — 보드 플래시 후 확인 대상.
